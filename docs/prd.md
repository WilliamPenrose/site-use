# site-use 产品需求文档

> 更新时间：2026-03-18

## 产品定位

通过对话操控你自己的 Chrome 浏览 Twitter——采集 KOL 内容、产出每日简报、随时追问和行动。

数据不出本地，不需要把账号交给第三方。针对特定站点写死操作流程，比通用 browser agent 更准确、更便宜。

---

## 目标用户

冷启动人群：AI/crypto 领域的独立开发者，自己也在 Twitter 上活跃，对 browser automation / AI agent 话题有天然兴趣。

为什么是这群人：
- 自己有痛点——每天刷 Twitter 追踪领域动态
- 能搞定安装——装个 npm 包、登录一次 Twitter 不是门槛
- 会传播——本身就在 Twitter 上发内容，用了觉得好会自发分享
- 对产品本身感兴趣——"确定性 workflow vs 通用 browser agent"这个技术选择本身就是话题

---

## 背景与动机

我关注 AI/agent 领域的最新进展，Twitter 是核心信息源之一。但 Twitter 免费 API 只能写不能读，获取内容只能手动浏览。

每天的流程是：刷 timeline → 发现新面孔 → 点进 profile 判断是否值得关注 → follow → 再逐个翻已关注 KOL 的最近动态 → 自己在脑子里综合归纳。这套流程每天要花不少时间，且大部分操作是机械性的。

市面上的通用 browser agent（如 browser-use）每个动作都调 LLM，成本高且不稳定。而"点 Follow 按钮""搜索关键词""翻 timeline"这些操作完全是确定性的，不需要 AI 参与。AI 应该花在"理解内容"上，不是"找到按钮在哪"。

核心需求：**采集已关注 KOL 的每日内容，提取有价值信息，综合产出每日简报，支持按需深挖。**

---

## 核心场景（MVP）

### 用户旅程全景

```
发现人 → 关注 → 每日简报 → 深挖 → 发现新的人 → …
```

用户的感受是：在用 OpenClaw 操作 Chrome 使用 Twitter。

### 场景 0：首次使用

第一次使用时需要两步：登录 Twitter + 选择追踪对象。

```
我："我想用你帮我追踪 Twitter 上的 AI 圈动态"
系统：（自动启动一个独立的 Chrome 窗口，不影响用户正在用的浏览器）
系统："请在这个浏览器窗口里登录你的 Twitter 账号。"
我：（登录完成）
系统：（自动进入 following 页面，滚动提取）
系统："你目前关注了 83 个人，我列出来了，你想追踪哪些人的动态？"
我："对，先用这些"
```

**成功标准：**
- 系统自动启动独立 Chrome 窗口，用户只需登录一次 Twitter
- 能自动读取用户的 Twitter following 列表
- 用户能看到浏览器在操作，知道系统在做什么
- 初始化（不含登录）< 3 分钟

### 场景 1：关注新 KOL

我通过各种渠道得知某人值得追踪，告诉系统去关注。我通常只知道人名，不知道 Twitter handle。

```
我："帮我关注 Andrej Karpathy"
系统：（Chrome 自动打开 Twitter 搜索页，搜索 "Andrej Karpathy"）
系统："找到了这个人：[头像] Andrej Karpathy @karpathy，前 Tesla AI 总监，2.1M 粉丝。确认关注吗？"
我："对"
系统：（Chrome 自动点击 Follow 按钮）
系统："已关注。"
```

**成功标准：**
- 能根据人名搜索，返回候选账号信息（头像、bio、粉丝数等）
- 经我确认后完成 follow
- 全流程 < 1 分钟

### 场景 2：每日简报

我已关注一批 AI 领域 KOL，每天想快速了解他们在聊什么。不是逐人摘要，而是跨人物的主题归纳和信号发现。

```
我："大家都在聊什么"
系统：（Chrome 自动打开 Twitter 首页 timeline，滚动提取推文，过滤广告和推荐内容——用户能看到整个过程）
系统：
"今天 3 个值得注意的信号：
 ① Karpathy 开始质疑 RAG 的长期方向，认为 long context 会取代大部分 RAG 场景
 ② 多人讨论 Claude 的 computer use 在企业场景的落地困难
 ③ MCP 协议出现第一个大规模生产部署案例"
```

**成功标准：**
- 能批量采集多个 KOL 的最新推文
- 简报按主题/信号组织，不是按人罗列
- 每条信号有来源归属（谁说的）
- 简报附带采集元信息（采集条数、覆盖 KOL 数 / 总关注数、时间范围），让用户能判断覆盖面
- 采集过程用户可见，知道系统在做什么
- 简报产出 < 5 分钟

### 场景 3：按需深挖

看到简报中某条信号感兴趣，想进一步了解。可以直接对话追问，也可以跳到 Twitter 原文。

```
我："展开讲讲 Karpathy 对 RAG 的质疑"
系统：给出原文内容、论据、相关讨论 + Twitter 原文链接

我："这个 @someone 观点挺好，帮我关注"  → 回到场景 1
```

**成功标准：**
- 能定位到具体推文原文
- 提供可点击的 Twitter 链接
- 支持在对话中展开讨论

## 产品边界

**做什么（MVP）：**
- 通过对话管理 Twitter 关注列表（搜索、follow）
- 批量采集已关注 KOL 的最新推文，产出跨人物每日简报
- 支持对简报内容追问、深挖原文
- 所有浏览器操作在用户本地 Chrome 中完成

**不做什么：**
- 不做内容发布——MVP 只读不写
- 不做定时任务——手动触发，不做后台调度
- 不做多站点——MVP 只做 Twitter，小红书是第二优先
- 不做历史趋势分析——只看当天内容，不跨天对比
- 不做通用 browser agent——不让 AI 看截图决定点哪里，而是针对特定站点写死操作流程

---

## 非功能需求

- **可靠性**：Twitter DOM 结构会变。选择器断裂时需要自愈能力，不能每次改版就手动维护。
- **浏览器隔离**：系统使用独立的 Chrome profile，不干扰用户日常浏览器。使用用户本地的 Chrome，不额外下载 Chromium。首次使用需用户手动登录 Twitter 一次，之后自动保持登录态。session 过期时通知用户重新登录。
- **反爬**：操作频率保持在正常人类使用水平。不做批量高频操作，不需要代理池等反反爬手段。
- **性能**：单次操作（follow、采集 20 条推文）< 30 秒。每日简报的完整采集 < 5 分钟。
- **操作可见性**：所有浏览器操作在用户自己的 Chrome 中执行，用户全程可见。这不是技术限制，而是产品特性——透明可控，数据不出本地，不需要把凭证交给第三方。
- **并发**：MVP 单 tab 顺序执行，但架构上不限制多 tab 并行，后续可按需扩展。

## 风险与假设

| 假设 | 风险 | 应对 |
|------|------|------|
| Twitter DOM 结构相对稳定，不会每天变 | 频繁改版导致选择器大面积失效 | 自愈选择器引擎降低维护成本；真的大改时需手动更新适配层 |
| 用户 Chrome 保持 Twitter 登录态 | session 过期导致操作失败 | 检测登录页特征，及时通知用户 |
| 正常频率操作不会触发 Twitter 反爬 | 被限流或封号 | 控制操作间隔，模拟人类节奏 |
| 从 timeline 采集能覆盖大部分 KOL 内容 | Twitter 算法过滤掉某些人的推文，导致遗漏 | 对特别关注的人，支持单独打开 profile 采集作为补充 |
| 用户本地安装了 Chrome | 用户用其他浏览器（Firefox、Edge 等）| MVP 只支持 Chrome，覆盖绝大多数用户 |
| 通用 browser agent 短期内成本高、准确率不够 | AI 操作浏览器的成本和准确率快速改善，site-use 的确定性 workflow 层失去存在价值 | 架构上 site-use 应该是可替换的——Twitter Skill 不关心数据怎么来的，只关心拿到结构化内容。哪天通用方案够好了，换掉 site-use，上层不用动 |

## 待明确问题

> 以下问题不影响产品方向，但技术设计时需要回答。

1. **搜索不到人怎么办**：场景 1 中，搜不到目标用户或候选都不匹配时，系统应该怎么反馈？是直接说"没找到"，还是建议换个关键词重试？
2. **自愈选择器引擎的深度**：PRD 要求"选择器断裂时能自愈"，但具体算法、权重、存储方案需要在技术设计中重点展开。这是可靠性承诺能否兑现的关键。
3. **KOL 列表管理**：场景 2 需要"批量采集已关注 KOL"，那 site-use 是维护自己的 KOL 列表，还是每次从 Twitter following 列表实时获取？两种方式的成本和一致性不同。
